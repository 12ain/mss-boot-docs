# 登录排障

登录失败通常来自认证状态、接口域名、菜单权限或浏览器会话。排查时先记录
请求路径、状态码和错误信息，不要在 Issue 或截图中暴露真实 token、密码、
数据库连接串或生产地址凭据。

## 快速定位

| 现象                   | 优先检查                           | 下一步                     |
| ---------------------- | ---------------------------------- | -------------------------- |
| 登录接口返回 `401`     | 账号密码、后端认证配置、用户状态   | 重新登录并查看后端登录日志 |
| 登录后很快跳回登录页   | token 过期、前端本地存储、系统时间 | 清理会话后重新登录         |
| 登录成功但菜单为空     | 角色菜单权限、菜单路由、接口权限   | 检查角色绑定和菜单返回值   |
| 前端页面打开但接口报错 | API 域名、代理、CORS、后端服务状态 | 对比前端环境变量和后端地址 |
| 同一个浏览器反复异常   | localStorage、cookie、旧构建缓存   | 使用无痕窗口复现并清理缓存 |

## 1. 登录接口返回 401

先确认请求是否到达了预期后端。开发环境可用：

```bash
curl -I http://127.0.0.1:8080/healthz
```

再检查登录请求：

- 请求路径是否是当前后端提供的登录接口。
- 账号是否存在且未被禁用。
- 密码是否仍是初始值或已经被维护者修改。
- 后端配置中的认证、JWT、数据库连接是否加载成功。
- 系统时间是否明显偏差，避免 token 签发后立即失效。

如果是公开 beta 环境，不要把真实响应 token、Cookie 或完整请求头贴到
公开 Issue。只保留状态码、接口路径、错误 key 和可脱敏截图。

## 2. Token 过期或会话混乱

常见表现是登录成功后短时间内又回到登录页，或刷新页面后所有接口变成
`401`。

排查步骤：

1. 退出登录后重新登录。
2. 清理当前站点的 localStorage 和 cookie。
3. 使用无痕窗口复现，排除旧会话干扰。
4. 检查后端 JWT 过期时间和签名配置是否在重启后变化。
5. 检查代理或网关是否移除了 `Authorization` 请求头。

## 3. 登录成功但菜单为空

登录成功只能说明认证通过，菜单为空通常是权限数据或前端路由数据问题。

优先检查：

- 当前用户是否绑定了正确角色。
- 角色是否绑定菜单权限。
- 菜单是否启用，路由地址是否存在。
- API 权限是否阻止了菜单接口。
- 前端是否使用了旧构建或旧路由配置。

本地调试时可以先用管理员账号验证，再切换到普通用户账号对比菜单返回值。

## 4. API 域名或代理错误

前端可访问但接口报错时，先确认前端实际请求的 API 域名。常见错误包括：

- 本地前端仍指向旧 beta/alpha API。
- Cloudflare 页面指向了开发环境 API。
- 本地代理未转发 `/admin/api/**`。
- 反向代理没有转发 `Authorization`、`Cookie` 或必要的 CORS 头。

开发环境建议先验证最小路径：

```bash
curl -I http://127.0.0.1:8080/healthz
curl -I http://127.0.0.1:8000
```

如果前端在本地启动，应对接 alpha/dev 后端；公开 beta 前端只应展示已经
通过冒烟测试的 beta API。

## 5. 浏览器缓存和旧构建

当只有某一个浏览器异常，而无痕窗口或另一台机器正常，优先按会话问题处理：

- 清理 localStorage 和 cookie。
- 强制刷新页面。
- 确认 Service Worker 或 CDN 没有缓存旧构建。
- 确认用户切换环境后没有复用旧 token。

## 提交 Issue 前的最小信息

请提供这些脱敏信息，便于维护者定位：

- 环境：local、alpha/dev、beta 或 prod。
- 前端页面路径。
- API 路径和状态码。
- 错误 key 或错误摘要。
- 是否可在无痕窗口复现。
- 是否仅某个账号、角色或菜单发生。

不要提交真实密码、token、cookie、生产用户数据、数据库连接串或集群配置。
